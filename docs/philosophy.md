# Philosophy

> How I think about building software.

<br/>

---

## Core Beliefs

### Software should solve meaningful problems.

The most elegant architecture in the world is worthless if it doesn't help anyone. Before writing a single line of code, I ask: *Who is this for? What problem does it solve? Will anyone's life be better because this exists?*

If the answer isn't clear, the code shouldn't be written yet.

---

### Build products, not projects.

A project demonstrates a skill. A product solves a problem. The difference is who you're building for.

Projects serve the builder. Products serve the user. I try to build products — software with a user, a purpose, and a reason to exist beyond the commit history.

---

### Simplicity is not the absence of complexity.

Simple software handles complex problems in ways that feel obvious to the user. That's hard. It requires understanding the problem deeply enough to find the simplest solution that actually works.

I value simplicity not because complex solutions are wrong, but because they're fragile, expensive to maintain, and hard to explain.

---

### Maintainability is a feature.

Clever code is debt. Clear code is an investment.

The best codebases I've worked with share a common trait: a new engineer can understand what's happening without a guided tour. That's not an accident — it's a design choice. Variable names, folder structure, documentation, test coverage — these are features, not afterthoughts.

---

### User experience is everyone's responsibility.

The best backend in the world is worthless behind a confusing interface. UX isn't a layer that gets added after the engineering is done — it's a constraint that shapes the engineering from the beginning.

I think about the user at every layer: API design, error messages, loading states, empty states, edge cases. The goal is software that feels obvious to use.

---

### Measure impact, not output.

Lines of code, commits per day, hours worked — these are vanity metrics. What matters is: *Did the software ship? Did users benefit? Did the problem get solved?*

Ship, observe, learn, iterate. Intuition is useful. Data is better.

---

### Stay curious.

The best engineers I've met never stopped being students. They read broadly, experiment freely, and ask questions that others assume have already been answered.

Technology moves fast. The only sustainable advantage is the willingness to keep learning.

---

## What This Means In Practice

- I start with the problem, not the technology.
- I write code that's easy to read, not just easy to write.
- I prefer shipping something imperfect over perfecting something unshipped.
- I document decisions, not just implementations.
- I test the things that matter, not the things that are easy to test.
- I think about the person who will maintain this code in two years. Often, that person is me.

---

## Influences

These are some of the ideas and thinkers that shape how I approach engineering:

- **Paul Graham** — On startups, building things, and thinking from first principles.
- **Dieter Rams** — "Good design is as little design as possible." Applies to software as much as industrial design.
- **The Unix Philosophy** — Do one thing well. Compose small, focused tools.
- **Basecamp / 37signals** — On building calm, sustainable software businesses.
- **Vercel / Guillermo Rauch** — On developer experience and making deployment invisible.
